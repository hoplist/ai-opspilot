# OpsPilot platform governance roadmap

## Goal

Make OpsPilot safe and simple for developers who do not understand Kubernetes,
GitOps, credentials, middleware, or observability wiring. The platform should
keep operational ownership while letting developers work with ordinary Git
pushes and natural-language OpsPilot requests.

The first governance stages are now in place. GitOps desired state has moved to
`tpo/deploy/gitops-manifests`. Argo CD portability can continue as a controlled
migration, but the live `argocd-core` package path should only be switched
after a render diff and Argo CD health verification.

The safer order is:

1. Build a credential ledger and repository permission boundary.
2. Publish a one-page developer push and release flow.
3. Template business-service GitOps output.
4. Refactor `argocd-core` into a portable Helm/Kustomize package.
5. Switch the live Argo CD Application path in a separate GitOps release.

## Current State

Current live paths still use the compatibility layout:

| Area | Current path | Purpose |
| --- | --- | --- |
| OpsPilot core | `tpo/platform/opspilot/opspilot-core` | Platform source released through the standard pipeline. |
| Runtime config | `tpo/platform/opspilot/opspilot-config` | GitLab-backed runtime config consumed by `opspilot-core` through git-sync. |
| Runtime skills | `tpo/platform/opspilot/opspilot-skills` | GitLab-backed skills consumed by `opspilot-core` through git-sync. |
| GitOps deploy state | `tpo/deploy/gitops-manifests` | Desired cluster state consumed by Argo CD. |
| Sandbox apps | `tpo/sandbox/devex/*` | Demo and validation application repositories. |
| Backups and ops assets | `tpo/ops/*` | Backup snapshots and manual ops holding assets. |

Target layout remains:

```text
tpo/
  apps/<org-group>/<project>/<service>
  platform/opspilot/<component>
  deploy/gitops-manifests
  shared/<template-repo>
  ops/<backup-or-helper>
  sandbox/<temporary-demo>
```

## Phase 1: Credential Ledger And Repository Boundary

### Credential Classes

Keep credentials grouped by who uses them and what they can affect:

| Class | Examples | Storage | Owner |
| --- | --- | --- | --- |
| Platform runtime | `OPSPILOT_GITLAB_TOKEN`, skills git token | Kubernetes Secret in `opspilot` namespace | Platform/Ops |
| CI automation | `GITOPS_TOKEN`, runner token | GitLab CI/CD variables or runner host config | Platform/Ops |
| Image pull | `gitlab-registry-pull`, node/containerd registry auth | Node/containerd config or Kubernetes Secret | Platform/Ops |
| Application runtime | MySQL/Redis/S3 service credentials | Service-scoped Kubernetes Secret | Platform/Ops, generated by OpsPilot |
| Temporary debug | Short-lived DB or API access | Time-limited account plus audit record | OpsPilot |
| Observability datasource | Prometheus, ELK/OpenSearch, OpenObserve, APISIX | Datasource Secret plus ConfigMap reference | Platform/Ops |

### Ledger Fields

Every credential should have a record, even when the value itself is not stored
in Git:

```yaml
name: opspilot-release-secrets
class: platform-runtime
environment: test
scope: node200/opspilot
usedBy:
  - opspilot-core
keys:
  - OPSPILOT_GITLAB_TOKEN
permissions:
  - read GitLab pipelines
  - read registry metadata
  - write GitOps only when rollback is explicitly confirmed
storage:
  type: kubernetes-secret
  namespace: opspilot
rotation:
  owner: platform
  interval: 90d
  lastRotated: ""
blastRadius:
  - release evidence
  - rollback
```

### Repository Boundary

Developers can push application code. They should not directly write deploy
state or platform credentials:

| Repository class | Developer writes? | Platform writes? | Notes |
| --- | --- | --- | --- |
| `tpo/apps/...` | Yes | Yes | Source code, tests, thin CI include, optional service declaration. |
| `tpo/deploy/gitops-manifests` | No | Yes | Argo CD source of truth. CI/OpsPilot updates it. |
| `tpo/platform/...` | No by default | Yes | Platform components and skills. |
| `tpo/shared/...` | No by default | Yes | CI, Dockerfile, and service templates. |
| `tpo/ops/...` | No | Yes | Backups and operational assets. |
| `tpo/sandbox/...` | Limited | Yes | Disposable demos and validation. |

## Phase 2: One-Page Developer Flow

Developers should only need this mental model:

```text
git clone app repo
-> write code
-> git push
-> GitLab CI checks/builds
-> image is pushed
-> GitOps is updated
-> Argo CD deploys
-> OpsPilot reports result and evidence
```

Minimum repository shape:

```text
README.md
src/ or cmd/
.gitlab-ci.yml
opspilot.service.yaml       # optional; OpsPilot should generate when missing
deploy/k8s/                 # generated starter output, not hand-crafted
```

The ideal `.gitlab-ci.yml` should only include a platform template:

```yaml
include:
  - project: tpo/shared/ci-templates
    file: /buildkit-gitops.yml
```

Developers should ask OpsPilot for service onboarding instead of learning
Dockerfile, BuildKit, GitOps paths, Argo CD Applications, or namespaces:

```text
帮我把这个仓库接入标准发布
帮我给这个服务加共享 MySQL
帮我查这次发布为什么失败
帮我回退到上一个健康版本
```

## Phase 3: Business-Service GitOps Templates

Business workloads should not hand-maintain large Deployment YAML files. The
service owner provides intent; OpsPilot generates the deploy surface.

Recommended service intent:

```yaml
name: skillshub-api
language: go
port: 8080
namespace:
  mode: project
resources:
  profile: small
probes:
  type: http
  path: /healthz
middleware:
  mysql:
    mode: shared
  redis:
    mode: none
storage:
  uploads:
    mode: hostpath
    path: /data/opspilot/hostpath
quality:
  apiSmoke: optional
```

Generated GitOps output should be predictable:

```text
clusters/<cluster>/apps/<group>/<project>/<service>/
  kustomization.yaml
  namespace.yaml
  resourcequota.yaml
  limitrange.yaml
  serviceaccount.yaml
  deployment.yaml
  service.yaml
  middleware-*.yaml
```

Namespace, resources, probes, image, and middleware should come from platform
catalog rules unless an explicit approved override exists.

## Phase 4: Portable Argo CD Core

`argocd-core` is currently long because it contains Argo CD itself: RBAC,
ConfigMaps, server, repo-server, controller, dex, redis, services, and network
policies. That is acceptable for a live bootstrap, but poor for migration.

Use Kustomize first because it matches the current GitOps repository and avoids
introducing Helm release state during the migration:

```text
platform/argocd/
  base/
  overlays/
    node200-test/
    other-cluster/
```

or use the upstream Helm chart with a small values file per cluster:

```text
clusters/<cluster>/platform/argocd/
  values.yaml
  application.yaml
```

Only these values should differ per cluster:

- GitLab URL and GitOps repository URL.
- Registry host.
- Argo CD exposure mode: NodePort, Ingress, or internal only.
- Resource limits.
- RBAC and SSO settings.
- Whether `prune` is enabled for platform self-management.

Current live path:

```text
clusters/test/argocd-core
```

Portable package path:

```text
platform/argocd/overlays/node200-test
```

Migration rule:

1. Render the old and new paths.
2. Compare resource identity and important specs.
3. Change `apps/argocd-core-application.yaml` source path only after the render
   diff is understood.
4. Keep `prune: false` for the first migration sync.
5. Delete the compatibility path only after Argo CD is `Synced` and `Healthy`.

## Multi-Cluster OpsPilot

OpsPilot can manage multiple clusters, but the safe model is not "copy all
kubeconfigs to every client." The safe model is:

```text
Client
-> OpsPilot API
-> cluster registry
-> per-cluster Kubernetes/Prometheus/log/release datasources
```

### Cluster Registry

Each cluster should be registered as a datasource bundle:

```yaml
clusters:
  node200-test:
    environment: test
    kubernetes:
      mode: in-cluster
      serviceAccount: opspilot-core
    prometheus:
      source: node200-k8s
    gitops:
      project: tpo/deploy/gitops-manifests
      pathPrefix: clusters/test
    argocd:
      namespace: argocd
    registry:
      host: 192.168.48.206:5050

  prod-a:
    environment: prod
    kubernetes:
      mode: remote
      secret: opspilot-cluster-prod-a
      kubeconfig: /var/run/opspilot/clusters/opspilot-cluster-prod-a/kubeconfig
      context: prod-a
    prometheus:
      source: prod-a
    logs:
      source: prod-elk
    gitops:
      project: tpo/deploy/gitops-manifests
      pathPrefix: clusters/prod-a
```

### Multi-Cluster Permissions

- Test clusters may allow read-only inspection plus controlled GitOps release
  actions.
- Production clusters should default to read-only evidence, with high-risk
  actions returning plans only.
- Remote kubeconfigs or service-account tokens must live in Kubernetes Secrets
  or an external secret manager, never on client machines.
- `opspilot-core` mounts remote kubeconfig Secrets under
  `/var/run/opspilot/clusters/<secret-name>/kubeconfig`; the CLI only sends
  `--cluster` and never receives kubeconfig content.
- Every command should accept or infer `--cluster`, and every result should show
  the active cluster.

### Runtime Configuration

The environment form used by the current implementation is metadata-only. It
registers where server-side kubeconfigs are mounted; it never stores the
kubeconfig content itself:

```text
OPSPILOT_CLUSTER_CATALOG="node200-test=environment:test,kubernetes:in-cluster,prometheus:node200-k8s,gitops_project:tpo/deploy/gitops-manifests,path:clusters/test,argocd_ns:argocd,registry:192.168.48.206:5050;prod-a=environment:prod,kubernetes:remote,secret:opspilot-cluster-prod-a,kubeconfig:/var/run/opspilot/clusters/opspilot-cluster-prod-a/kubeconfig,context:prod-a,prometheus:prod-a,logs:prod-elk,gitops_project:tpo/deploy/gitops-manifests,path:clusters/prod-a"
```

Remote kubeconfig Secret example:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: opspilot-cluster-prod-a
  namespace: opspilot
type: Opaque
stringData:
  kubeconfig: |
    # kubeconfig content is managed outside Git
```

`opspilot-core` should mount the Secret at:

```text
/var/run/opspilot/clusters/<secret-name>/kubeconfig
```

The CLI only sends:

```text
--cluster prod-a
```

It must not read, copy, or print kubeconfig contents.

### Multi-Cluster Commands

Future command shape:

```text
opspilot clusters list
opspilot check cluster --cluster node200-test
opspilot check service skillshub-api --cluster prod-a
opspilot metrics filesystems --cluster all
opspilot release status --service skillshub-api --cluster node200-test
```

## Implementation Order

1. Add docs and ledger format.
2. Move natural-language intent parsing behind the OpsPilot backend.
3. Add read-only `credentials catalog` and `clusters catalog` commands.
4. Move service recommendations and skill routing fully server-side.
5. Add credential registration plans for app middleware and observability
   datasources.
6. Generate business GitOps from service intent files.
7. Add multi-cluster datasource registry and cluster-aware execution.
8. Refactor `argocd-core` into portable base/overlays or Helm values.
