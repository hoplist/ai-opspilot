# 2026-06-01 Optional API quality checks

## Goal

Add an optional post-deploy quality evidence step to OpsPilot. It should let a
service repository define lightweight API smoke and latency checks, then let
OpsPilot run them after a service is deployed and attach the result to release
evidence for AI troubleshooting.

This is optional evidence. Missing test config, missing GitLab access, missing
runner image, or missing Kubernetes Job permission must not block core
Kubernetes inspection, release status, rollback, logs, metrics, or AI fix
planning.

## Scope

- Generate a starter `.opspilot/quality.yaml` during service onboarding.
- Add repository preflight visibility for `.opspilot/quality.yaml` as a
  warning-level optional item, not a blocker.
- Add an OpsPilot quality runner command that can execute simple HTTP checks
  inside the cluster and emit machine-readable JSON.
- Add backend endpoints to submit and inspect quality Jobs for configured
  release services.
- Add CLI commands:
  - `quality run service <name>`
  - `quality status service <name>`
- Add latest quality evidence to release status without changing the release
  health decision.
- Add capability detection for the optional quality runner.

## First Version

The first version intentionally stays small:

- HTTP smoke checks by method, path, expected status, and timeout.
- Simple latency threshold using observed request duration.
- A short-lived Kubernetes Job runs the checks from inside the service
  namespace.
- The Job uses the OpsPilot image and `opspilot quality runner`.
- Results are read from Job logs and exposed as release evidence.

The first version does not add load testing, full functional testing, active
security scanning, service mesh requirements, or mandatory quality gates.

## Repository Contract

Service repositories may include:

```text
.opspilot/quality.yaml
```

Starter content generated by OpsPilot:

```yaml
quality:
  enabled: true
  optional: true
  baseURL: http://<service>.<namespace>.svc.cluster.local:<port>
  smoke:
    timeoutSeconds: 3
    latencyP95Ms: 1000
    endpoints:
      - name: health
        method: GET
        path: /health
        expectStatus: 200
```

If this file is missing or disabled, OpsPilot reports `skipped` quality
evidence and continues normally.

## Execution Model

```text
GitLab Runner
-> BuildKit
-> GitLab Registry
-> GitOps
-> Argo CD
-> OpsPilot release status
-> optional OpsPilot quality Job
-> release evidence / AI evidence pack
```

OpsPilot creates Jobs only when explicitly requested through `quality run`.
Later automation can call this after Argo CD reports Healthy, but the quality
step remains non-blocking unless a future policy explicitly changes that.

## Kubernetes Job Model

- Namespace: service namespace.
- Name: `<service>-quality-<timestamp>`.
- Image: `OPSPILOT_QUALITY_RUNNER_IMAGE`.
- Command: `/usr/local/bin/opspilot quality runner`.
- TTL: `OPSPILOT_QUALITY_JOB_TTL_SECONDS`, default 3600.
- Deadline: `OPSPILOT_QUALITY_DEADLINE_SECONDS`, default 120.
- Resources: small fixed requests and limits.

Expected RBAC addition:

- `batch/jobs`: `get`, `list`, `watch`, `create`.

If the RBAC is missing, `quality run` returns `unavailable` evidence rather
than failing the whole release path.

## User Flow

```powershell
opspilot onboard repo tpo/devex/demo/demo-api --write
opspilot release service demo-api --trigger --output human
opspilot release status --service demo-api --output human
opspilot quality run service demo-api --output human
opspilot quality status service demo-api --output evidence
```

## Safety

- No destructive Kubernetes operation is required.
- Jobs are short-lived and TTL-cleaned by Kubernetes.
- Quality checks run with tight CPU/memory limits.
- Missing optional evidence is surfaced explicitly but not promoted to a
  release blocker.
- Active DAST, ZAP, k6 load testing, and functional suites are deferred.

## Validation Plan

- Unit test quality config parsing and HTTP runner behavior.
- Unit test CLI routing for `quality run/status`.
- Unit test onboarding generation of `.opspilot/quality.yaml`.
- Unit test repository preflight marks quality config as optional.
- Run `go test ./opspilot/...`.
- Publish through the standard flow:

```text
node206 GitLab
-> node206 GitLab Runner
-> BuildKit rootless image build
-> GitLab Registry
-> GitOps repository
-> node200 Argo CD automatic deployment
```

## Rollback

Rollback follows the existing OpsPilot release rollback path by reverting the
GitOps image tag for `opspilot-core`. Since the quality feature is optional,
disabling `OPSPILOT_QUALITY_ENABLED` also removes runtime impact without code
rollback.
