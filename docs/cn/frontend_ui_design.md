# RCA 工作台前端产品设计文档（AI 可读版）

## 1. 产品定位

RCA 页面不是传统监控看板，也不是纯日志页，而是一个面向运维排障的调查工作台。

目标是让用户按下面顺序完成任务：

1. 先快速看到当前最可信的结论
2. 再确认支撑该结论的高价值证据
3. 再决定下一步动作
4. 最后再进入日志、事件、监控等深层证据页

页面应服务两类场景：

- 值班排障：几分钟内找到最可能原因和建议动作
- 事后复盘：按时间线回看根因、证据和动作建议

## 2. 设计原则

### 2.1 核心原则

页面必须符合真实排障顺序，而不是按数据来源堆砌模块。

优先级固定为：

1. 结论
2. 高危信号
3. 根因与建议动作
4. 时间线
5. 运行上下文
6. 原始日志与事件
7. 外部跳转入口
8. 历史调查

### 2.2 交互原则

- 用户进入页面后，首屏必须能读懂当前对象发生了什么
- 调查输入区不应占据首屏太大面积
- 原始日志表格不能抢占第一视觉焦点
- 右侧侧栏应始终承担“下一步去哪里”的导航作用
- 页面应参考 Langfuse 工作台：主信息清晰、卡片分组明确、信息密度高但不拥挤

### 2.3 视觉原则

- 使用浅色、长时间阅读友好的分析型工作台风格
- 不做“大屏监控风”，而是做“排障决策台”
- 风险状态使用明确颜色，但不要整页过红
- 面板分组清楚，层级比装饰更重要

## 3. 用户任务流

用户在页面上的典型行为是：

### 3.1 值班排障路径

1. 输入 namespace / pod / 时间范围
2. 发起调查
3. 阅读一句话结论
4. 查看关键指标摘要
5. 查看根因排序和建议动作
6. 查看关键时间线
7. 必要时下钻日志 / 事件 / 外部 Dashboard

### 3.2 复盘路径

1. 打开最近调查记录
2. 先看结论和风险信号
3. 看时间线和上下文
4. 再看原始日志与事件
5. 结合建议动作做复盘总结

## 4. 页面总体布局

采用三栏桌面布局：

- 左侧：导航栏
- 中间：主工作区
- 右侧：侧栏入口区

页面结构如下：

```text
左侧导航栏 | 中间主工作区 | 右侧侧栏
```

## 5. 页面结构定义

## 5.1 左侧导航栏

### 作用

- 建立“工作台”感
- 提供页面锚点
- 固定品牌与产品身份
- 支持在长页面中快速跳转

### 包含内容

1. 品牌区
2. 页面导航锚点
3. 可选的设计说明 / 使用提示卡片

### 导航锚点建议

- 调查输入
- 调查摘要
- 结论
- 关键指标
- 根因分析
- 时间线
- Pod 上下文
- 日志证据
- K8s 事件
- 历史调查

### 设计要求

- 固定在页面左侧
- 当前滚动区域高亮
- 收起后保留图标导航
- 不喧宾夺主，弱于主区

## 5.2 中间主工作区

中间主区必须严格按“排障顺序”组织，而不是按数据类型组织。

推荐顺序如下：

1. Hero
2. 调查输入区
3. 调查摘要卡
4. 一句话结论
5. 关键指标摘要
6. 根因分析网格
7. 运行上下文区
8. 日志证据
9. K8s 事件
10. 原始调查提示词

## 5.3 右侧侧栏

### 作用

右侧栏不是补充信息区，而是行动入口区。
它回答的是：我接下来该看什么、点哪里。

### 顺序

1. Current Incidents
2. 调查对象列表
3. Dashboards 直达
4. 最近调查记录

### 设计要求

- sticky 固定
- 卡片高度适中
- 每张卡片能快速点击跳转
- 必须比主区信息弱一级，但始终在用户视野中

## 6. 组件设计规范

## 6.1 Hero

### 目标

定义页面语气，让用户知道这是一个“先结论、再证据、再动作”的排障工作台。

### 内容

- 小标签：AI RCA / Auto Inspection
- 主标题：一句工作原则式标题
- 副标题：说明使用方式
- 右上角：资源看板 / ELK 日志平台 / Prometheus 等外部入口

### 推荐文案

主标题：

先看结论，再看证据，再决定动作

副标题：

这个页面按真实排障顺序组织，帮助你在几分钟内定位最重要的信息。当前判断、关键指标、证据链和可执行动作会按优先级展示。

### 设计要求

- 标题明显大于其他区域
- 副标题不超过两行
- 外部入口按钮视觉弱于主标题
- Hero 不要太高，避免挤压核心内容

## 6.2 调查输入区

### 功能目标

用于发起调查，但不能成为首屏视觉中心。

### 字段

- Namespace
- Pod
- Workload
- Range Hours
- Question
- Query Hint
- Use AI

### 操作

- 开始调查
- 加载最近一次
- 保留用户最近输入状态

### 设计要求

- 使用紧凑卡片
- 表单分两行排布
- 默认不比 Hero 更高
- 发起按钮明确
- “Use AI” 应与提交动作靠近，形成一组

### 交互要求

- 输入变化保留本地状态
- 点击加载最近一次后自动填充表单
- 发起调查时显示明确 loading 状态
- 调查成功后滚动定位到摘要区或结论区

## 6.3 调查摘要卡

### 目标

帮用户确认“当前看到的是哪一次调查”。

### 固定四张卡片

- 调查 ID
- 证据来源
- 目标对象
- 存储状态

### 设计要求

- 四张等宽卡片
- 信息简短，不堆长文本
- 状态信息可以带徽标
- 存储状态建议显示“已落库 / 部分落库 / 未持久化”

## 6.4 一句话结论

### 目标

用户不看任何表格，也能先知道当前最强判断。

### 内容结构

- 标题：一句话结论
- 主文案：当前最可信判断
- 可选：置信度 / 严重级别 / 影响对象

### 推荐结构

```text
Pod langfuse/langfuse-clickhouse-shard0-0 持续重启；
当前最强信号指向：容器启动阶段内存不足或 limit 过低，导致 OOM。
```

### 设计要求

- 大字号
- 占主区上方
- 使用更强视觉容器
- 可加左侧风险色条
- 结论必须比“关键指标摘要”更显眼

## 6.5 关键指标摘要

### 目标

把会直接影响判断的指标先抽出来，而不是让用户自己看 Prometheus 原始图。

### 固定展示指标

- Restart Total
- Restart Increase
- Waiting Reason
- Last Terminated Reason
- Memory Request / Limit
- CPU Request / Limit

### 设计要求

- 6 张小卡片
- 支持风险色条
- 高危项突出显示
- 支持解释型文案，比如“可疑时间窗内增量”“最近终止原因”

### 风险优先级建议

- OOMKilled、CrashLoopBackOff：高危
- Restart Increase 高于阈值：中高危
- Memory Request 明显偏低：高危
- CPU 信息通常次于内存信息

## 6.6 根因分析区

该区域是页面核心之一，推荐使用 2x2 网格。

### 四块内容

1. 根因排序
2. 建议动作
3. 需要人工确认
4. 关键时间线

### 6.6.1 根因排序

#### 目标

给出当前最可能原因及证据。

#### 内容结构

- 原因标题
- 置信度
- 支撑证据列表

#### 示例

- Memory limit too low or workload memory spike causes OOM during startup
- 置信度 92%
- 支撑证据：
  - waiting reason = CrashLoopBackOff
  - last terminated reason = OOMKilled
  - memory req / limit 偏低

### 6.6.2 建议动作

#### 目标

给出 3 到 5 条最可能有效的下一步动作。

#### 内容要求

- 动作必须可执行
- 动作顺序按推荐优先级排序
- 尽量是“运维动作”，不是泛泛建议

#### 示例

1. 提高 clickhouse 容器 memory request 和 limit
2. 检查启动期 merge / 恢复过程是否带来瞬时内存峰值
3. 查看上一个容器退出前日志确认 OOM 前关键行为

### 6.6.3 需要人工确认

#### 目标

明确告诉用户哪些判断需要人工补充确认，避免 AI 过度自信。

#### 示例

- 确认是否存在连续快速重启导致的残留 socket 或多进程启动问题
- 确认最近是否调整了镜像、探针或配置

#### 设计要求

- 使用提示型卡片
- 视觉弱于“建议动作”，但不能隐藏

### 6.6.4 关键时间线

#### 目标

按时间顺序快速给出高价值事件链。

#### 内容

- 时间
- 事件类型
- 简短描述

#### 示例

- 02:02 Readiness probe failed
- 01:57 BackOff restarting failed container
- 01:56 Container terminated with OOMKilled

#### 设计要求

- 时间线必须简洁
- 每条不超过 2 到 3 行
- 支持点击跳到底部事件表或日志表

## 6.7 运行上下文区

推荐两张卡片并列：

- Pod 快照
- Prometheus 上下文

### 6.7.1 Pod 快照

#### 内容

- Pod 名称
- Phase
- Node
- Owner
- 主容器状态
- Restart 次数
- Waiting Reason
- Exit Code
- 关键 requests / limits

#### 设计要求

- 结构化信息优先
- 不用大段 JSON
- 重点字段加标签展示
- 状态有颜色

### 6.7.2 Prometheus 上下文

#### 内容

- CPU
- Memory
- Restart Total
- Restart Increase
- Waiting Reason
- 可选附加信号

#### 目标

把 Prometheus 结果转成能看懂的摘要，而不是图表堆叠。

## 6.8 日志证据区

### 目标

展示 AI / 调查流程筛出的高价值日志，而不是把全量日志直接堆上来。

### 设计要求

- 放在结论之后，不要抢首屏
- 默认展示异常日志片段
- 支持 current / previous 容器区分
- 支持横向滚动
- 支持“展开全部日志视图”

### 表格列建议

- 时间
- 容器
- 日志来源
- 类型（current / previous）
- 内容摘要

### 交互建议

- 支持按关键字过滤
- 支持只看 error / oom / probe 失败
- 支持跳转到 ELK 网关/业务日志视图

## 6.9 K8s 事件区

### 目标

帮助用户理解状态迁移、探针失败、调度问题、BackOff 过程。

### 表格列建议

- Timestamp
- Type
- Reason
- Object
- Message

### 设计要求

- 使用标准宽表格
- Message 支持换行
- 可按 Reason 快速过滤
- 提供“跳到时间线”的联动能力

## 6.10 原始调查提示词

### 目标

给 AI 调查过程提供可追溯性，方便评估质量，但不是首要模块。

### 要求

- 默认折叠或弱化显示
- 放在页面最底部
- 标记“仅供复盘与质量校验”

## 7. 右侧侧栏设计

## 7.1 Current Incidents

### 目标

告诉用户当前还有哪些高风险对象需要关注。

### 内容

- 对象名
- 一句话风险摘要
- 风险等级
- 数量徽标

### 设计要求

- 高优对象置顶
- 支持点击跳转调查
- 明确当前对象是否就在列表中

## 7.2 调查对象列表

### 目标

展示本次调查的核心对象入口。

### 内容

- pod / workload 名称
- 摘要文案
- 删除或关闭入口

## 7.3 Dashboards 直达

### 目标

提供下一步外跳入口。

### 推荐入口

- 总览 Dashboard
- RCA Dashboard
- 日志视图
- 事件视图

### 设计要求

- 每项带简短说明
- 不直接展示长 URL
- 使用“打开”或“跳转”动作按钮

## 7.4 最近调查记录

### 目标

快速回到最近调查，不让用户重新输入。

### 内容

- 对象名
- 摘要
- 时间
- 风险等级

## 8. 组件层级与排版规范

## 8.1 版心

- 主内容区宽度优先保证可读性
- 不要拉得过宽导致文字难读
- 桌面端建议主区和侧栏之间有稳定间距

## 8.2 卡片规范

- 统一圆角
- 统一阴影层级
- 统一标题、说明、内容间距
- 卡片标题风格统一，避免混乱

## 8.3 间距

推荐统一 spacing 体系，例如：

- 模块间距：24
- 卡片内边距：20
- 小组件间距：12 到 16

## 8.4 字体层级

建议层级：

- Hero 标题：最大
- 一句话结论：次大
- 模块标题：中
- 卡片主值：中到大
- 说明文字：小

## 9. 状态与颜色规范

### 9.1 状态颜色

- Critical：红
- Warning：橙 / 黄
- Healthy：绿
- Neutral：灰

### 9.2 颜色使用原则

- 风险色只用于真正关键状态
- 不要大面积使用高饱和红色
- 按“点状强调”而不是“整块警报”处理

### 9.3 适合强调的对象

- OOMKilled
- CrashLoopBackOff
- Restart Increase
- Unhealthy
- BackOff

## 10. 响应式规范

## 10.1 桌面端

- 左导航、主区、右侧栏同时存在
- 右侧栏 sticky
- 主区双列分析网格可用

## 10.2 平板端

- 左导航收缩
- 右侧栏下移
- 2x2 根因网格可转 2x1

## 10.3 手机端

- 以主调查链路为主
- 右侧栏内容并入主区后部
- 表格尽量提供摘要视图而非完整宽表

## 11. 页面存在问题与改版要求

以下是当前页面需要改进的重点：

### 11.1 输入区过大

问题：调查输入区占首屏过多。
要求：压缩输入区高度，使结论更快进入首屏。

### 11.2 结论不够强

问题：一句话结论虽然存在，但视觉上没有形成绝对中心。
要求：提升字号、容器权重和间距，让它成为首屏主角。

### 11.3 关键指标与根因区关系不够紧

问题：当前结论、关键指标、根因之间的连续性不强。
要求：形成“结论 → 指标 → 根因与动作”的紧密阅读链。

### 11.4 时间线价值高但位置不够突出

问题：时间线目前只是普通卡片。
要求：让时间线在根因分析区中更显眼，成为第二层重点。

### 11.5 右侧栏入口价值高但描述弱

问题：侧栏有入口，但信息密度和风险提示不足。
要求：增加状态徽标、数量和简短文案。

### 11.6 表格区阅读成本高

问题：日志和事件表格信息多，但缺少过滤和固定列。
要求：

- 支持固定时间列
- 支持 reason / container 快速过滤
- 默认只显示高价值片段

### 11.7 中英文混用和遗留文案问题

要求：

- 全面清洗文案
- 统一中文优先
- 保留英文仅用于技术字段值

## 12. 参考风格要求（Langfuse 风格映射）

本页面参考 Langfuse 工作台，但不是照搬。

应借鉴的点：

1. 模块层次清楚
2. 卡片信息密度高但不乱
3. 主区聚焦核心任务
4. 侧边入口明确
5. 页面整体更像“分析工作台”而不是“表单后台”

不要照搬的点：

1. 不要把 RCA 页面做成通用 LLM 平台页
2. 不要过度追求营销感 hero
3. 不要让 trace 风格替代排障顺序

## 13. 给 AI / 前端生成页面的明确要求

请按以下要求生成或优化页面：

### 13.1 布局要求

- 使用三栏桌面布局：左导航 + 中主区 + 右侧栏
- 中主区必须按真实排障顺序排列模块
- 右侧栏 sticky，作为入口与跳转区
- 输入区紧凑化，不抢首屏
- 一句话结论必须在首屏高可见区域

### 13.2 组件要求

需要包含以下组件：

- Hero
- 调查输入区
- 4 张摘要卡
- 一句话结论卡
- 关键指标摘要卡组
- 根因分析 2x2 网格
- Pod 快照卡
- Prometheus 上下文卡
- 日志证据表格
- K8s 事件表格
- 右侧侧栏卡组
- 原始调查提示词折叠卡

### 13.3 视觉要求

- 浅色分析型工作台风格
- 统一卡片样式
- 有风险色条但不过度告警
- 重点强调结论、风险、动作
- 保持长时间阅读舒适度

### 13.4 交互要求

- 支持发起调查与加载最近一次
- 支持页面锚点导航
- 支持日志和事件的快速过滤
- 支持跳转外部 Dashboard
- 支持历史调查回看
- 支持保留本地输入状态

### 13.5 内容优先级要求

渲染顺序必须遵循：

1. 结论
2. 风险信号
3. 根因与动作
4. 时间线
5. 运行上下文
6. 原始证据
7. 侧栏入口
8. 历史内容

## 14. 输出目标

该页面优化后的最终目标是：

- 用户进入页面 3 秒内知道最可能出了什么问题
- 用户 10 秒内知道先看哪几个证据
- 用户 30 秒内知道下一步该做什么
- 页面同时适合值班使用和事后复盘

## 15. 适合直接喂给 AI 的简版提示词
